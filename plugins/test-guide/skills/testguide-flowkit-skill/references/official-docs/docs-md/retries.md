# Retries configuration

This guide explains how to configure HTTP request retries for tools used in flow.kit.

Retries can help to make flows more robust against **temporary** network issues (too many requests, connection resets) or intermittent server problems.

Tip

Retry configuration is optional. Only configure retries if you experience intermittent request failures (e.g. HTTP 429 - Too Many Requests or connection resets). If `retries_configuration` is not set, the tool/client uses its default retry behavior.

## Configuration object

Retries are configured using `flow_kit.tools.retries_configuration.RetriesConfiguration`.

It currently supports the following options:

- `total: int | None`
  - Total number of retries that may be performed.
  - If set to `None`, the default behavior of the client implementation is used.
- `all_methods: bool` (default: `False`)
  - Danger

    Before enabling `all_methods`, please read the section [`Retry on all HTTP methods`](#retry-on-all-http-methods) below.

  - If `True`, retries are allowed for **any** HTTP method (including `POST`, `CONNECT`, `PATCH`).

  - If `False`, the default behavior of the client implementation is used (e.g. `GET`, `HEAD`, `PUT`, `DELETE`, `OPTIONS`, `TRACE`).

## Where to configure retries

Retries are passed to tool initialization blocks via their optional parameter `retries_configuration`.

The following init blocks currently support this parameter:

- [`CodebeamerInit`](Block%20documentation/tools/codebeamer/index.md)
- [`EtmInit`](Block%20documentation/tools/etm/index.md)
- [`JamaInit`](Block%20documentation/tools/jama/index.md)
- [`PolarionInit`](Block%20documentation/tools/polarion/index.md)

## Usage

### Basic example

The following example shows how to configure retries for ETM initialization. The same pattern applies to the other init blocks.

    import ...
    from flow_kit.tools.retries_configuration import RetriesConfiguration


    def get_flow() -> Flow:
        return (
            FlowBuilder()
            .add_block_with(
                etm := EtmInit().with_label("Configure ETM connection"),
                # ... URL & credentials setup ...
                Assign(EtmInit.PAR__RETRIES_CONFIGURATION).to_static_value(
                    RetriesConfiguration(total=3)
                ),
            )
            .build()
        )

### Retry on all HTTP methods

Danger

Setting `all_methods=True` enables retries for all methods and can cause repeated `POST`/`PATCH`/`DELETE` requests. Only use this if the remote system and your use case can handle it.

This is especially critical for **non-idempotent** operations: if a request is sent successfully but the HTTP client does not receive the response (e.g. too many requests, connection reset), the retry may execute the same operation again. Possible consequences in automated flows include for example:

- **Duplicate creations** (e.g. the same item is created multiple times)
- **Repeated updates** (potentially overwriting newer data)
- **Unintended deletions** or “too many” deletions (depending on the API endpoint semantics)

Recommendation: clarify with the operating IT / system owner whether the target system (and any intermediaries like reverse proxies/load balancers) can safely handle retries for writing requests.

    import ...
    from flow_kit.tools.retries_configuration import RetriesConfiguration


    def get_flow() -> Flow:
        return (
            FlowBuilder()
            .add_block_with(
                CodebeamerInit().with_label("Configure Codebeamer connection"),
                # ... URL & credentials setup ...
                Assign(CodebeamerInit.PAR__RETRIES_CONFIGURATION).to_static_value(
                    RetriesConfiguration(total=4, all_methods=True)
                ),
            )
            .build()
        )

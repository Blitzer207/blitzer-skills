# Introduction

The flow.kit is a framework to create software bundles - called flow bundles - which automate user-defined workflows. These flow bundles are self-contained i.e. they can automate your user-defined workflow without external dependencies.

## Creating a flow bundle for your user-defined workflow

The flow.kit contains a library of flow blocks. You build the automation for your user-defined workflow by using flow blocks from this library in an artifact called flow definition. You build the representation of your user-defined workflow with a flow definition by selecting, configuring and connecting flow blocks in a graph. The following image shows a graph of an exemplary workflow consisting of 6 blocks. The arrows show the connections between the blocks modelling the block dependencies.

![sample_flow_overview](res/introduction/execution_engine_1_before_execution.png)

The flow.kit together with your flow definition forms a flow bundle. Such a flow bundle is a self-contained executable for the automation of your user-defined workflow.

## How to represent your workflow by building a flow definition from flow blocks?

The flow definition is built from a selection of configured and connected flow blocks.

A flow block is the representation for an action in the workflow. The flow.kit contains a large set of flow blocks for different actions (e.g. for interaction with test.guide and Jira) that you can use in your workflow. Every flow block has the following elements:

- **Action**: Most flow blocks have a predefined specific action which is already implemented for you. In addition, user-defined flow blocks with custom action implementations are possible.
- **Parameters**: Every flow block has a custom set of configurable parameters. You customize the function of the flow block by setting the parameters. For parameter values you can use static values, references to results of other flow blocks and custom user expressions.
- **Result**: The result provides the output value of the executed action of the flow block. You can use the result as parameter value for other flow blocks.
- **Required dependencies**: You can define explicit dependencies to other flow blocks. With this feature you influence the execution order flow blocks in addition to the data flow from results to parameters of flow blocks.

Flow blocks have dependencies on one another in the form of explicit dependencies (required dependencies) and implicit dependencies (flow block results provided as parameters of other flow blocks). Dependencies of flow blocks mean that all depending flow blocks need to be executed successfully before the flow block itself is executed.

Specific information on how to build the flow definition for your workflow can be found in the section [How to Build a Flow Definition](how_to_build_a_flow_definition.md#how-to-build-a-flow-definition).

## Execution of flow definitions

When executing the flow bundle the flow blocks in the flow definition are run by the execution engine of the flow.kit. The execution engine runs the flow blocks individually taking care of the individual dependencies between the blocks. If no dependency is defined between blocks they are executed in any order.

[TABLE]

The following images illustrate the execution and the flow block states of a flow definition for an exemplary workflow consisting of 6 blocks:

**Step 1**

Block 1 is ready for execution as it has no unfinished depending blocks. The blocks 2-6 are not ready for execution.

![sample_flow_execution_step1](res/introduction/execution_engine_2_block_1_ready.png)

**Step 2**

Block 1 has been executed and is now in state FINISHED. Blocks 2 and 3 are ready for execution. As there is no dependency between them both blocks can run in any undefined order.

![sample_flow_execution_step2](res/introduction/execution_engine_3_block_1_run.png)

**Step 3**

Block 2 was executed and is now in state ABORTED. A block gets state ABORTED after the execution if an error occured during its execution.

![sample_flow_execution_step3](res/introduction/execution_engine_4_block_2_run.png)

**Step 4**

Block 4 gets state SKIPPED because it cannot be executed as the dependency block 2 has state ABORTED.

![sample_flow_execution_step4](res/introduction/execution_engine_5_block_4_skipped.png)

[]() **Step 5**

Block 6 gets state SKIPPED because it cannot be executed as well (dependency block 4 has state SKIPPED).

![sample_flow_execution_step5](res/introduction/execution_engine_6_block_6_skipped.png)

**Step 6**

Block 3 was executed and is now in state FINISHED. As the dependencies of block 5 (only block 3) were executed successfully block 5 is now ready for execution.

![sample_flow_execution_step6](res/introduction/execution_engine_7_block_3_run.png)

**Step 7**

Block 5 was executed successfully and is now in state FINISHED.

The execution of the flow definition is now finished because all blocks are in one of the three final states FINISHED, ABORTED or SKIPPED.

![sample_flow_execution_step7](res/introduction/execution_engine_8_block_5_run.png)
